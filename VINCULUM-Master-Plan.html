Section 01

The Vision

Every K–12 math benchmark gets a visual bridge — an interactive tool that makes the abstract concrete.

The VINCULUM Math Solutions is a repository of standalone, self-contained HTML web apps — one for every math benchmark across multiple standards frameworks. Each app is a purpose-built visual tool that combines interactive manipulatives, guided practice with scaffolding, and simple feedback-driven games.

Unlike platforms like Desmos, GeoGebra, or Khan Academy that bolt interactivity onto content delivery, VINCULUM tools are built from the standard outward. Each tool exists because a specific benchmark demands it, and each tool is designed to address the exact misconceptions and bridging concepts that benchmark requires.

Core Principles

🎯

Standard-First Design

Each app is born from a benchmark. The standard defines the interaction, not the other way around. Misconceptions data and ALDs drive the scaffolding.

✨

Aesthetic Excellence

Dark-themed, high-contrast, color-coded interfaces following the VINCULUM design system. Every tool looks and feels like it belongs in the same family.

🧩

Three Modes Per Tool

Explore (manipulative), Practice (guided with scaffolds + feedback), Play (simple game reinforcement). Every benchmark, three modes.

🔗

Ecosystem Connected

Every tool has a standards ID, metadata tags, and an API-ready manifest so the Lesson Digestor can find and serve the right tool automatically.

Section 06

Hub Design (Central Launchpad v2)

Upgrading your existing VINCULUM Math Solutions Central Launchpad into the standards-driven Hub.

Your current Launchpad already has the card-based launcher and category sidebar. The v2 upgrade adds three critical features: standards-based navigation, multi-framework filtering, and the manifest system that lets the Digestor query it programmatically.

Hub Navigation Layers

1

Grade Selector

Horizontal bar: K | 1 | 2 | 3 | 4 | 5 | 6 | 7 | 8 | ALG1 | GEOM | ALG2 | PRE-C. Click a grade, see only that grade's tools.

2

Domain Filter

Sidebar updates per grade. Grade 4 shows: OA, NBT, NF, MD, G. Grade 8 shows: NS, EE, F, G, SP. Click a domain to filter cards.

3

Standard Search

Type "4.NBT.1" or "place value" or "MN.4.2.1.1" and instantly find the matching tool. Cross-framework search built in.

Tool Card Design (Enhanced)

Each card in the Hub shows: tool name, icon, version, status badges (Explore/Practice/Play), aligned standards list (toggleable between CCSS/MN/TEKS), and a one-line description pulled from the manifest. Clicking a card opens the tool in fullscreen. A "Teacher Info" toggle reveals the misconceptions data and differentiation strategies from your STEM TC database.

Section 07

Lesson Digestor Integration

How the Digestor pulls from the Hub to serve the right tools automatically.

The Lesson Digestor takes a teacher's uploaded lesson, parses it with AI to identify which standards are being taught, then queries the Hub's manifest.json to find and serve the exact matching visual tools. Here's the flow:

1

Upload

Teacher uploads a lesson plan, textbook page, or worksheet.

2

Parse

AI identifies the standards being addressed (CCSS, MN, TEKS codes).

3

Match

Queries manifest.json for tools that map to those standards. Returns URLs.

4

Serve

Presents the matching tools with recommended modes and differentiation tips.

Manifest Schema (manifest.json)

{ "tools": [ { "id": "equation-lab", "name": "VINCULUM PRO: Equation Lab", "version": "5.5", "standards": { "ccss": ["8.EE.8", "HSA-REI.6", "HSA-REI.11"], "mn": ["8.2.4.4", "9.2.3.2"], "teks": ["A.5A", "A.5C"] }, "grade": ["8", "ALG1"], "domain": ["Expressions & Equations", "Algebra"], "modes": ["explore", "practice", "play"], "toolType": "graphing-engine", "url": "/tools/8/equation-lab.html", "description": "High-performance symbolic graphing engine for systems of equations", "misconceptions": [ "Students confuse the solution of a system with the x-intercept", "Students think parallel lines have 'zero solutions' means y=0" ], "differentiation": { "emergent": "Provide pre-set equations, focus on reading intersection point", "proficient": "Students create own systems and predict intersection", "advanced": "Explore systems with no solution or infinite solutions" } }, ... ] }
Section 08

Tech Stack Options

Choosing the right App Engine for building, hosting, and serving 400+ interactive HTML tools.

Your existing tools are built with Google Apps Script, which was great for prototyping. For the full ecosystem, here are the options ranked by fit for your project:

★

Option A: Static HTML + GitHub Pages

RECOMMENDED. Each tool is a single self-contained HTML file. No server, no framework, no dependencies. GitHub Pages hosts for free. Claude builds each file. You already have a GitHub account.

FREESIMPLESTOFFLINE-READY
Pros: Zero hosting cost, works offline, each tool is independent, easy to version, easy for Claude to build
Cons: No server-side logic (but you don't need it for visual tools)
B

Option B: Static HTML + Netlify/Vercel

Same as Option A but with automatic deployments from Git, custom domains, and serverless functions if you eventually need them (e.g., for the Digestor AI backend).

FREE TIERAUTO-DEPLOY
Pros: Same simplicity + continuous deployment + edge CDN
Cons: Slight learning curve for setup
C

Option C: React SPA (Single Page App)

All tools built as React components inside one application. Shared design system, shared state, smooth navigation. More complex to build but most polished UX.

POLISHEDCOMPLEX
Pros: Shared code, consistent design, app-like navigation
Cons: Higher complexity, build step required, harder to maintain per-tool
D

Option D: PWA (Progressive Web App)

Option A or B with a service worker for offline capability. Students install it on Chromebooks and use it without internet. Ideal for schools with unreliable Wi-Fi.

OFFLINEINSTALLABLE
Pros: Works offline, installable on devices, app store publishable
Cons: Adds service worker layer (manageable)

Recommended Stack

Start with Option A (Static HTML + GitHub Pages) for immediate builds, then upgrade to Option D (PWA wrapper) once you have 20+ tools. This gives you the simplest path to start building tools immediately while keeping the door open for offline/installable capability later.

Section 09

Build Phases

A phased rollout from your existing tools to the full K–12 ecosystem.

Phase 1 — Foundation (Weeks 1–2)

Repo Setup + Design System + Hub v2

Establish the repository, shared design system, and upgraded Hub with standards navigation.

  • Create GitHub repo with folder structure
  • Extract VINCULUM design system into shared CSS (dark theme, colors, typography, card styles)
  • Build Hub v2 (index.html) with grade selector, domain filter, and standard search
  • Create manifest.json schema and populate with existing tools
  • Port existing GAS tools (Equation Lab, Place Value, Coord Grid) to standalone HTML
Phase 2 — Core Tool Engines (Weeks 3–6)

Build the 10 Most-Used Tool Types

Focus on the tool engines that cover the most benchmarks across grades.

  • Number Line Engine (configurable K–8)
  • Fraction Engine (bars, area models, circle models)
  • Balance / Equality Model
  • Array / Area Model Builder
  • Algebra Tile Engine
  • Data Display Builder
  • Probability Simulator
  • Measurement Lab
  • Counting & Cardinality Lab
  • Ratio / Proportion Engine
Phase 3 — Standards Population (Weeks 7–10)

Configure Tools Per Benchmark

Take each core engine and create configured versions for specific grade-level benchmarks.

  • Number Line → K counting, 2nd addition, 3rd fractions, 6th integers, 8th irrationals
  • Build the standards crosswalk database (CCSS ↔ MN ↔ TEKS)
  • Add misconceptions and differentiation data from STEM TC
  • Add Practice mode (guided problems with scaffolding) to each tool
  • Add Play mode (simple game with feedback) to each tool
Phase 4 — Advanced Tools (Weeks 11–14)

HS Tools + 3D + Speciality Engines

  • Geometry Construction Lab
  • 3D Geometry Visualizer (Three.js)
  • Function Transformer
  • Unit Circle / Trig Lab
  • Upgrade existing Formula Interface and Vertex Ledger
Phase 5 — Digestor Integration (Weeks 15–16)

Connect the Lesson Digestor to the Hub

  • Build Digestor's standard-identification engine
  • Implement manifest.json query API
  • Create "Suggested Tools" panel in Digestor UI
  • Add teacher-facing differentiation recommendations per tool
  • End-to-end test: upload lesson → identify standards → serve tools
Phase 6 — Polish & PWA (Weeks 17–20)

PWA Wrapper + Offline + Final Polish

  • Add service worker for offline capability
  • Create PWA manifest for Chromebook installation
  • Accessibility audit (keyboard nav, screen readers, color contrast)
  • Mobile/tablet responsive testing
  • Performance optimization (lazy loading tools)
  • Launch documentation and teacher onboarding guide
